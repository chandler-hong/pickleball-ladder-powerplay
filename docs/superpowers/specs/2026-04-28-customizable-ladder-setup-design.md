# Customizable Traditional Ladder Setup — Design

## Background

The Traditional Ladder mode in `js/ladder.js` is currently fixed at **20 players on courts 2, 3, 7, 8, 9**. Two constants drive this:

- `LADDER_PLAYER_COUNT = 20`
- `LADDER_COURTS = [2, 3, 7, 8, 9]` (low-to-high; winners move up the array)

Round Robin mode already supports a customizable number of players, courts, and per-court court numbers. This design extends Traditional Ladder mode with the same flexibility, plus a new manual court-assignment UI.

## Goals

1. Let the user choose any reasonable number of courts (1–10) for ladder mode.
2. Let the user customize the court numbers (e.g., 2, 3, 7, 8, 9) instead of the hardcoded list.
3. Update the setup message to reflect the user's chosen configuration.
4. Let the user manually assign players to courts via drag-and-drop, in addition to the existing random gender-balanced assignment.

## Non-goals

- Allowing `players ≠ 4 × courts` (no byes/sit-outs in ladder mode). Ladder requires every player on a court each round; supporting byes would require redesigning the up/down movement logic.
- Allowing court reconfiguration mid-ladder. Once a ladder is active, setup is locked; the user must `Reset Ladder` to change court count or numbers.
- Drag-to-reorder within a court (partner pairing within a court is still computed by `bestPairing` from partnership history).

## UI changes

### Setup card (replaces current static "Ladder Setup" message)

Two inputs in a row, styled like Round Robin's setup:

- **Number of Courts** — `<input type="number" min="1" max="10" value="5">` (editable, primary input)
- **Number of Players** — read-only computed display (`<span>20 players</span>`), updated live as `4 × courts`

The dynamic message below the inputs replaces the current hardcoded text:

> `<N> players on courts <C1>, <C2>, ... — winners move up, losers move down. Drag players in the Court Assignments card below to set initial positions, or click "Re-shuffle" for a random gender-balanced setup.`

The court list in the message is shown bottom-to-top to match the round narrative ("winners move up").

### Court Numbers card (new)

Mirrors Round Robin's `Court Numbers` card. One `<input type="number">` per court, in **top-to-bottom-of-the-ladder order** (left-most input = top court / highest, right-most = bottom court / lowest). Defaults to `N, N-1, ..., 2, 1`. Validates per-input range (1–99) and uniqueness across the row.

Internally, we always store/compute `LADDER_COURTS` as low-to-high (current convention), so the rendered top-to-bottom inputs are reversed when read.

### Players card

Unchanged behaviour, but the player count grows/shrinks dynamically with `4 × courts`. M/F toggle, name auto-detection, live substitution all work as before. When `numCourts` changes, the player grid is rebuilt:

- Increase: keep existing player names/genders, append empty rows for new slots.
- Decrease: drop player slots from the end. If any of the dropped slots had a non-empty name, show a `confirm()` dialog before applying (see "Court count change" below for details).

### Court Assignments card (new)

Placed **after** the Players card, **before** the controls. Renders one vertical column per court, top-to-bottom of the ladder (top court = leftmost column). Each column header shows the court label (e.g., "Court 9 (Top)") and contains 4 player chips.

Each chip:

- Shows the player's name (live-updates from the Players card; falls back to "Player N" if blank).
- Shows a tiny M/F badge (also live).
- Is `draggable="true"`; HTML5 drag-and-drop with no external library.

Drag behaviour:

- Drag chip A onto chip B → **swap** their court assignments. This preserves the invariant of 4 players per court.
- Drop on an empty area or invalid target → no-op (chip returns to its original slot).
- Visual feedback: dragging chip becomes semi-transparent; drop target highlights on `dragover`.

Buttons on this card:

- **Re-shuffle** — re-runs the existing gender-balanced random allocation (`ladderInitialAllocation` logic, refactored to be assignment-only).

The card is always visible and always pre-populated. The default `manualAssignment` is generated by `randomGenderBalancedAssignment` and stored in `ladderConfig` whenever:

- The user first enters ladder mode (and no persisted `manualAssignment` exists).
- `Number of Courts` changes (full re-shuffle, see below).
- The user clicks `Re-shuffle`.

The user can either drag to customize or just click `Start Ladder` to use the current state.

### Controls card (existing)

Buttons stay: `Reset Ladder`, `Fill Random Names`, `Start Ladder`. No new buttons here.

### During an active ladder

When `ladderState !== null` (a ladder is in progress), all setup inputs (`Number of Courts`, court numbers, manual assignment chips) are disabled/greyed out. The user must `Reset Ladder` to change them. Player names remain editable for live substitution (existing behaviour).

## Data model changes

### New state (in-memory + persisted)

```js
ladderConfig = {
  numCourts: 5,                 // 1-10
  courtNumbers: [9, 8, 7, 3, 2],  // low-to-high (canonical), length = numCourts
  manualAssignment: [           // per-court player slots, low-to-high to match courtNumbers
    [0, 1, 2, 3],   // court at index 0 (lowest, e.g. 2): player slots 0,1,2,3
    [4, 5, 6, 7],   // court at index 1 (e.g. 3)
    ...
  ],
};
```

Player slot indices are stable across name edits (matching the existing `lp0..lp19` pattern). The `manualAssignment` array is a permutation of `0..numPlayers-1` partitioned into `numCourts` arrays of length 4.

### Existing constants → derived values

- `LADDER_PLAYER_COUNT` → replaced with `ladderConfig.numCourts * 4` everywhere it's used.
- `LADDER_COURTS` → replaced with `ladderConfig.courtNumbers` (low-to-high).

Both names are kept as accessor functions (`getLadderCourts()`, `getLadderPlayerCount()`) for minimal churn at call sites. Once a ladder is started, the existing `ladderState` snapshots them into the state object so historical rounds keep their config even if the user later resets and reconfigures.

### Persistence

`ladderConfig` is added to the `LADDER_STORAGE_KEY` blob in `saveLadderState` / `restoreLadderState`. Backward compatibility: when restoring an older blob without `ladderConfig`, default to the legacy `numCourts: 5, courtNumbers: [2,3,7,8,9]`.

`ladderState` (when active) also embeds a copy of `numCourts` and `courtNumbers` so historical round rendering uses the config that was in effect when the ladder started.

## Behaviour

### Initial assignment

The current `ladderInitialAllocation` does two things:

1. Distribute genders evenly across courts (random gender-balanced assignment).
2. Compute initial team pairings via `bestPairing`.

We split this into two functions:

- `randomGenderBalancedAssignment(genders, numCourts)` → returns the per-court player-slot arrays. Used by **Re-shuffle** and as the default when the user opens ladder mode for the first time.
- `pairTeamsForAssignment(courtPlayers, genders, preferMixed, partnerHistory)` → for each court, runs `bestPairing` to produce `{ teamA, teamB }`. Used at `ladderStart` time after the user's (possibly manual) assignment is finalized.

`ladderStart` reads `ladderConfig.manualAssignment` directly (no random fallback at start time, since the panel is always pre-populated). The state is then frozen into `ladderState` as today.

### Movement (unchanged)

`ladderProcessMovement` already operates on `LADDER_COURTS` as a generic array; once we swap that for `ladderConfig.courtNumbers`, no logic changes are needed. Re-pairing per court continues via `bestPairing`.

### Re-shuffle

Calls `randomGenderBalancedAssignment(getLadderGenders(), numCourts)`, writes the result to `ladderConfig.manualAssignment`, and re-renders the Court Assignments card.

### Court count change

When `Number of Courts` changes from `M` to `N`:

1. If `N < M` and dropping slots `4N..4M-1` would discard player names that the user has entered, show a `confirm()` dialog: *"Reducing the court count will remove the last `4(M-N)` players (`<names>`). Continue?"* If declined, revert the input to `M` and abort.
2. Rebuild the Players grid to size `4N`. Existing player names/genders preserved by slot index for slots that still exist.
3. Rebuild the Court Numbers card to `N` inputs. Existing values preserved by index where the index is still valid; new slots default to `N, N-1, ...` skipping any number already in use.
4. Always re-run `randomGenderBalancedAssignment` to produce a fresh `manualAssignment`. Any prior manual placement is discarded; a small banner in the Court Assignments card notes: *"Re-shuffled because court count changed."* The banner clears on the next user-initiated change in the card.

### Court number change

Pure label change. Each court's player set stays put; only the displayed court labels update. Update validation to flag duplicate court numbers (mirrors round-robin behaviour).

### Player name / gender change

Re-render the Court Assignments card (chips' name/badge updates) but leave the assignment itself untouched. This matches the live-substitution model already used elsewhere.

## Validation

In `ladderValidate`, add:

- `numCourts` is 1–10.
- `courtNumbers`: each is 1–99, all unique.
- All player slots `0..4*numCourts-1` must have non-empty names (already covered by adapting the existing loop bound).
- Existing duplicate-name check still applies.

## Files touched

- `index.html` — Setup card markup, Court Numbers card markup, Court Assignments card markup, dynamic message; remove the hardcoded "20 players on courts 2, 3, 7, 8, 9" line.
- `js/ladder.js` — replace `LADDER_PLAYER_COUNT` / `LADDER_COURTS` constants with config accessors; refactor `ladderInitialAllocation` into the two helpers above; add `ladderConfig` state; build/teardown helpers for the new cards; drag-and-drop wiring; resize/re-shuffle logic; update `ladderValidate`, `ladderStart`, `saveLadderState`, `restoreLadderState`, `ladderReset`, the rendering functions (`renderLadderCurrentRound`, `renderLadderHistory`, `renderLadderLeaderboard`).
- `js/state.js` — no functional change, but `setMode('ladder')` may need to call a new `buildLadderSetup()` to populate the new cards (currently it only calls `buildLadderPlayerGrid`).
- `css/styles.css` — styling for chips, drag-over highlight, dragging state, and the Court Assignments card layout. Reuse existing classes where possible (e.g., `.player-input`, `.gender-toggle`).
- `README.md` — update the Traditional Ladder Mode bullet to reflect the new flexibility.

## Out of scope (not changed)

- Round Robin mode UI/behaviour.
- Score tracking, leaderboard, history, exports.
- Algorithmic improvements to `bestPairing` or `ladderProcessMovement`.
- Tests directory: this work is UI-heavy and the existing `tests/` only covers round-robin scheduling. No new tests are added in this change. (A future change could add Node-runnable unit tests for `randomGenderBalancedAssignment` and the manual-assignment plumbing if desired.)

## Open questions resolved during brainstorming

- Player/court relationship: tied; players auto-derived from courts.
- Court number ordering in inputs: top-to-bottom of the ladder (top court first).
- Manual assignment UI: drag-and-drop chips per court.
